# Obsidian 项目说明书风格

## 目标

给人看的说明书必须像"项目说明书", 不是日志, 不是碎片笔记, 也不是过于朴素的占位模板.

风格要求:
- 有编号结构
- 大纲清晰
- 主文档短而能扫
- FAQ 高密度
- 细节只在必要时外链

## 默认文件数

默认只维护:
- 1 篇主文档

只有主文档明显装不下时, 才允许再增加:
- 1 篇 `reference` 文档

不要默认拆成很多文档.

## 主文档推荐结构

```md
# 项目名

## 一, 项目概述
### 1.1 核心功能
### 1.2 当前状态
### 1.3 关键限制

## 二, 安装和配置
### 2.1 获取或进入项目
### 2.2 关键目录结构
### 2.3 基础配置

## 三, 运行和验证
### 3.1 启动方式
### 3.2 访问地址 / 使用入口
### 3.3 验证方式

## 四, 关键流程
### 4.1 标准流程
### 4.2 关键脚本 / 页面 / 命令
### 4.3 易错点

## 五, 常见问题
| 问题 | 解决方案 |
| --- | --- |

## 六, 详细参考
- [[项目名.reference]]
```

## 写法要求

- 标题尽量像说明书, 不要像聊天摘要.
- 优先写"现在有效的结论", 不写过程流水账.
- FAQ 尽量写成"问题 -> 解决方案"表格.
- 命令, 地址, 路径, 配置项可以直接出现.
- 主文档只放高频内容, 低频细节放 `reference`.

## 禁止事项

- 不要写成时间线日志
- 不要堆太多"补充说明"
- 不要为每个小点单独开新文档
- 不要把 AI 内部碎记忆原样贴给人看
